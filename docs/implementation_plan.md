# План разработки: GTNH OC Water Line Control v2

На основе анализа проблем оригинального скрипта, мы разработаем версию **v2**, которая будет полностью модульной, сетевой (клиент-серверной), отказоустойчивой и простой в настройке.

---

## 1. Архитектурные роли (Разделение логики)
Скрипт будет поддерживать 3 роли, настраиваемые через `setup.lua`:
1. **`standalone` (Автономный)** — один компьютер делает всё: управляет транспозерами, опрашивает машины и рисует интерфейс (как в классической версии).
2. **`daemon` (Сервер-контроллер)** — ставится непосредственно у линии очистки воды. У него нет монитора. Он опрашивает машины, управляет транспозерами и рассылает состояние сети (`state.lua`) по сетевой карте (модему) раз в секунду. Слушает входящие команды переключения тиров.
3. **`gui` (Клиент-монитор)** — устанавливается в удобном месте на базе с 3x2 монитором. У него нет физического подключения к трубам или очистной установке. Он просто принимает состояние по сети от сервера и рисует красивый интерфейс, отправляя команды нажатия кнопок обратно на сервер.

---

## 2. Пошаговый план реализации

### Этап 1: Разработка каркаса и модулей связи
* Создание `config.lua` с поддержкой роли (`role`) и сетевого порта (по умолчанию `2424`).
* Разработка `lib/network.lua` для сериализации/десериализации сетевых пакетов, отправки логов и команд (`waterline_state`, `waterline_cmd`, `waterline_log`).
* Изменение `main.lua` как единой точки входа, которая в зависимости от роли запускает:
  * Только контроллеры + трансляцию сети (`daemon`).
  * Только отрисовку GUI + приём сетевых пакетов (`gui`).
  * Всё вместе локально (`standalone`).

### Этап 2: Повышение отказоустойчивости (Hot-Reconnect и pcall)
* Добавление защищенных вызовов (`pcall`) при каждом опросе компонентов (машин GregTech, транспозеров, ME-интерфейсов).
* Если компонент отключается:
  * Программа НЕ вылетает.
  * Состояние тира переходит в `"DISCONNECTED"`.
  * Каждые 5-10 секунд запускается фоновый поток автоматического поиска и повторного подключения к периферии.

### Этап 3: Интерактивный мастер настройки (`setup.lua`)
* Разработка удобного псевдографического интерфейса для настройки программы прямо в игре:
  * Выбор роли компьютера (Standalone / Daemon / GUI).
  * Автоматический поиск подключенных транспозеров с возможностью "подмигнуть" (подать редстоун-сигнал на стороны), чтобы игрок понял, какой транспозер к какому сундуку/люку подключен.
  * Привязка UUID транспозеров к конкретным тирам (T3–T8) и запись в `registry.lua`.
  * Установка вебхуков Discord. 
  * Меню полного удаления программы с диска (`uninstall`).

### Этап 4: Разделение потоков отрисовки и опроса
* Использование механизма таймеров OpenComputers (`event.timer`), чтобы разделить опрос машин (медленный, раз в 1-2 секунды) и отрисовку GUI / обработку клавиш (быстрый, по событиям).
* Это сделает интерфейс мгновенно реагирующим на нажатия клавиш без фризов.

### Этап 5: Оптимизация парсера сенсоров GregTech
* Оптимизация регулярных выражений в `lib/gt-sensor-parser.lua`.
* Добавление кэширования результатов, чтобы не парсить строки чаще необходимого.
* Поддержка альтернативных языковых шаблонов для стабильной работы на русскоязычных и англоязычных серверах.

### Этап 6: Интеграция с Git и деплой
* Настройка `update.lua` для загрузки всех файлов v2 с GitHub на компьютер OC.
* Автоматический пуш готового кода в репозиторий `Kwazzi44/GTNH-OC-Water-Line-Control-main-v2.git`.
